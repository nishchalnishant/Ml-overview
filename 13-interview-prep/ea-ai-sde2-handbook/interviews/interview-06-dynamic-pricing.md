## Part 4 — Expected Assumptions

- **Approach:** Contextual Bandits or Uplift Modeling. A pure classifier (P(buy | discount)) doesn't optimize for revenue/LTV directly.
- **Constraints:** Discounts can only be from a discrete set: {0%, 10%, 25%, 50%}.
- **Batching:** Promotions are generated daily in a batch pipeline and loaded into a fast key-value store (Redis) for the game client to fetch on login.

## Part 5 — High-Level Solution

```
  Data Warehouse (Historical A/B Test Data)
       │
       ▼
  Model Training (Contextual Bandit / Uplift Model)
  ┌────────────────────────────────────────────────────────┐
  │ Features: Player spend history, playtime, wallet size  │
  │ Arms (Actions): 0%, 10%, 25%, 50% discount             │
  │ Reward: 30-day LTV (revenue)                           │
  └────────────────────────────────────────────────────────┘
       │
       ▼ (Daily Airflow Batch Job)
  Batch Inference
  ┌────────────────────────────────────────────────────────┐
  │ Predict expected reward for each arm for every player  │
  │ Select the arm that maximizes reward (with exploration)│
  └────────────────────────────────────────────────────────┘
       │
       ▼
  Redis (Key: player_id, Value: discount_tier)
       │
       ▼
  Game Store API (Fetches discount on player login)
```

**Core ML Component:** A Contextual Bandit using Thompson Sampling or an Epsilon-Greedy approach on top of an XGBoost regressor that predicts 30-day revenue given (User Features + Discount Arm).

---

## Part 6 — Step-by-Step Implementation

### Step 1: Formulate the Reward
- Reward = Total revenue generated by the user in the 30 days following the promotion.
- *Wait, if we use 30-day LTV, feedback is delayed by 30 days.* For daily bandit updates, we might need a proxy metric (e.g., 3-day revenue + expected wallet depletion). Let's stick to a batch-trained model updated weekly.

### Step 2: Model Choice (Contextual Bandit via Regression)
- Train $N$ separate XGBoost regressors (one for each discount tier/arm).
- Input: Player Features.
- Target: 30-day Revenue.
- Inference: Run the player through all 4 models. Pick the arm that yields the highest predicted revenue.

### Step 3: Exploration Strategy (Epsilon-Greedy)
- For 90% of players, give them the optimal predicted discount (Exploitation).
- For 10% of players, give them a random discount (Exploration) to gather unbiased data for next week's model training.

### Step 4: Feature Engineering
- Current premium currency balance (high balance = more likely to buy).
- Days since last purchase.
- Historical price sensitivity (have they only bought on sale?).
- Engagement metrics (daily active minutes).

## Part 7 — Complete Python Code

```python
"""
dynamic_pricing_bandit.py - Batch Contextual Bandit for Store Promotions
"""
import logging
import numpy as np
import pandas as pd
import xgboost as xgb
import redis
from typing import List, Dict

logging.basicConfig(level=logging.INFO)
logger = logging.getLogger(__name__)

# Discrete discount arms
ARMS = [0.0, 0.10, 0.25, 0.50]
EPSILON = 0.10 # 10% exploration

class PricingBandit:
    def __init__(self):
        # We hold one model per arm
        self.models: Dict[float, xgb.Booster] = {}
        
    def load_models(self, model_paths: Dict[float, str]):
        for arm, path in model_paths.items():
            model = xgb.Booster()
            model.load_model(path)
            self.models[arm] = model
            
    def predict_best_arm(self, features_df: pd.DataFrame) -> pd.DataFrame:
        """
        Calculates expected revenue for all arms and selects the max,
        applying epsilon-greedy exploration.
        """
        player_ids = features_df['player_id'].values
        X = features_df.drop(columns=['player_id'])
        dmatrix = xgb.DMatrix(X)
        
        # 1. Predict expected revenue for each arm
        predictions = {}
        for arm in ARMS:
            # Predicts 30-day LTV
            predictions[arm] = self.models[arm].predict(dmatrix)
            
        # Convert to matrix: shape (num_players, num_arms)
        pred_matrix = np.column_stack([predictions[arm] for arm in ARMS])
        
        # 2. Find best arm (Exploitation)
        best_arm_indices = np.argmax(pred_matrix, axis=1)
        best_arms = [ARMS[idx] for idx in best_arm_indices]
        
        # 3. Apply Epsilon-Greedy Exploration
        final_arms = []
        is_exploring = []
        
        for i in range(len(player_ids)):
            if np.random.rand() < EPSILON:
                # Explore
                chosen_arm = float(np.random.choice(ARMS))
                final_arms.append(chosen_arm)
                is_exploring.append(True)
            else:
                # Exploit
                final_arms.append(best_arms[i])
                is_exploring.append(False)
                
        # 4. Result DataFrame
        result_df = pd.DataFrame({
            'player_id': player_ids,
            'assigned_discount': final_arms,
            'is_exploration': is_exploring
        })
        
        # Add expected revenues for logging/analytics
        for i, arm in enumerate(ARMS):
            result_df[f'exp_rev_{arm}'] = pred_matrix[:, i]
            
        return result_df

def push_to_redis(result_df: pd.DataFrame, redis_client: redis.Redis):
    """Pushes the personalized discounts to Redis for the Game API."""
    pipeline = redis_client.pipeline()
    for _, row in result_df.iterrows():
        key = f"store_promo:{row['player_id']}"
        # Store discount and expiry (24 hours)
        pipeline.setex(key, 86400, row['assigned_discount'])
        
        # Execute in chunks to avoid blowing up Redis memory/network
        if len(pipeline) >= 1000:
            pipeline.execute()
            
    if len(pipeline) > 0:
        pipeline.execute()
    logger.info("Successfully pushed promotions to Redis.")

def run_batch_job():
    # In production, fetch from Snowflake
    dummy_data = pd.DataFrame({
        'player_id': ['p1', 'p2', 'p3'],
        'wallet_balance': [1000, 50, 0],
        'days_since_purchase': [2, 30, 100],
        'playtime_7d': [1400, 120, 0]
    })
    
    bandit = PricingBandit()
    # Mock model loading
    # bandit.load_models({0.0: "m0.json", 0.1: "m1.json", 0.25: "m2.json", 0.5: "m3.json"})
    
    # Mocking predictions for compilation sake
    bandit.models = {arm: type('Mock', (), {'predict': lambda self, x: np.random.rand(len(dummy_data)) * 100})() for arm in ARMS}
    
    results = bandit.predict_best_arm(dummy_data)
    print(results)
    
    # r = redis.Redis(host='localhost', port=6379)
    # push_to_redis(results, r)

if __name__ == "__main__":
    run_batch_job()
```

### Game Store API (Serving)
- A lightweight Go or FastAPI service reads from Redis. Latency is < 5ms.
- If Redis is down, or key is missing, default to `0.0` discount (Fail safe to protect revenue).

## Part 11 — Scaling Discussion

| Axis | Strategy |
|------|----------|
| **10+ Million Players** | Using raw Pandas will OOM. Use PySpark `pandas_udf` to distribute the XGBoost inference across a cluster, or use Dask. |
| **Redis Memory** | Storing 10M keys in Redis takes memory. Use Redis Hashes instead of plain strings to optimize memory overhead, or only push keys for active players. |
| **Model Number Explosion** | 4 arms = 4 models. If we have 20 arms (items x discounts), we have 20 models. Switch from separate models to a single model where `Discount_Arm` is an input feature. |

---

## Part 12 — Tradeoffs

| Decision | Tradeoff |
|----------|----------|
| Contextual Bandit vs Uplift Modeling | Bandits are easy to implement with continuous exploration. Uplift modeling (T-learners, X-learners) explicitly models the *causal* effect of the discount but is harder to implement and evaluate. |
| Single Model vs Multi-Model (per arm) | Multi-model isolates the signal perfectly but requires training $N$ models. Single model is easier to maintain but might ignore the discount feature if other features (like playtime) are too dominant. |
| Epsilon-Greedy vs Thompson Sampling | Epsilon-Greedy wastes 10% of traffic on completely random choices (suboptimal). Thompson Sampling balances exploration/exploitation much more efficiently using probability distributions, but is mathematically complex to implement for XGBoost. |

---

## Part 13 — Alternative Approaches

1. **Uplift Modeling (Causal ML):** We don't care about absolute revenue, we care about *incremental* revenue. If a "Whale" player will spend $100 anyway, giving them a 50% discount drops revenue to $50. Uplift models specifically optimize for the difference between Treatment and Control.
2. **Deep Reinforcement Learning (DQN):** Instead of a static 30-day horizon, use an RL agent that takes sequential actions (pricing over time) to maximize long-term reward. Too complex for v1, but the "holy grail" of dynamic pricing.

## Part 14 — Failure Scenarios

| Failure | Impact | Mitigation |
|---------|--------|-----------|
| Bad Model Release | Model assigns 50% discount to everyone | Add strict business rule guards before Redis push: "Max 5% of DAU can receive the 50% discount." |
| Redis down | Players cannot see store | Store API must degrade gracefully to 0% discount (base price) if Redis timeouts. |
| Feedback Loop / Bias | Model only learns about 0.25 arm | Epsilon exploration guarantees we collect data for all arms. Without it, the model becomes biased and collapses. |

## Part 15 — Debugging

**Symptom:** Finance reports that Store Revenue dropped by 15% after deploying the model, even though the model predicted a 10% increase.

**Debugging steps:**
1. Check the Exploration data. Did the 10% holdout group (random discounts) outperform the Exploitation group? If yes, the model's logic is fundamentally flawed.
2. Check for Cannibalization: Did we measure *immediate* revenue instead of *30-day* revenue? Players might have bought the discounted pack but stopped buying normal packs.
3. Check the prediction distributions: Are the XGBoost predictions heavily skewed or miscalibrated? (e.g., predicting $1000 LTV for a new player).
4. Verify the Redis sync. Did the Store API actually serve the discounts we predicted, or was there a bug (e.g., passing 0.50 as a price instead of a discount)?

## Part 17 — Production Improvements

1. **Move to Causal Inference (Uplift):** Use `EconML` or `CausalML` libraries to implement a Meta-Learner (X-Learner). This explicitly prevents giving discounts to "Sure Things" (Whales).
2. **Thompson Sampling:** Replace Epsilon-Greedy with a Bayesian approach (e.g., Bayesian Linear Regression on top of deep embeddings) to explore more intelligently.
3. **Budget Constraints:** Add a Linear Programming solver after inference to maximize revenue subject to a constraint (e.g., "Do not give out more than $100,000 worth of discounts today").

## Part 19 — Ideal Answers

**Q1 (Whale cannibalization):**
> "This is exactly why predicting absolute revenue (LTV) is dangerous. We must predict *incremental* revenue. If $E[Rev | 0\%] = \$100$, and $E[Rev | 50\%] = \$80$, the arm with the highest expected absolute value is 0%. We pick 0%. However, if the model has high variance, it might mistakenly predict the 50% arm is $105. To fix this, we should use Uplift Modeling, which explicitly models $E[Rev | Treatment] - E[Rev | Control]$. If the uplift is negative, we don't treat."

**Q2 (Legal constraints / Price Discrimination):**
> "We pivot from *personalized pricing* to *personalized item targeting*. The price of the 'Gold Pack' remains identical for everyone. Instead, our model decides *which items* to feature in the 'Just for You' store slot. We show the Gold Pack to Player A, and the Bronze Pack to Player B. Same bandit architecture, but the arms are the items, not the discounts."

**Q3 (Delayed Reward):**
> "We need to build a Surrogate Metric (proxy metric). We train a secondary machine learning model that takes 3 days of user behavior after the promotion and predicts their 30-day LTV. We then use this *predicted* 30-day LTV as the reward for our Bandit model. This allows us to close the feedback loop in 3 days instead of 30."
