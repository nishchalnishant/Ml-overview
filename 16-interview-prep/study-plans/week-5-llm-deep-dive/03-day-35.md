# Day 35: Applied & Production LLMs + Mock Interview

## Why This Topic Comes Here

The final day of the deep dive shifts from mechanism to application: multimodal architectures, context-length scaling, and the production discipline (LLMOps) that differs from classical MLOps. It closes with a mock interview to stress-test retrieval of the whole week under pressure, mirroring the Week 4 final-prep format.

---

## Multimodal Models

- Common pattern: a modality-specific encoder (vision transformer, audio encoder) projects into the same embedding space as text tokens, then a shared transformer backbone processes the combined sequence.
- Key design question: early fusion (combine modalities before the backbone) vs. late fusion (separate processing, combine near the output) — early fusion allows richer cross-modal reasoning but costs more compute and is harder to train stably.

## Context Window Extension

- Techniques: RoPE interpolation/extrapolation (NTK-aware scaling), ALiBi (extrapolates by design), sliding-window attention, and retrieval-augmented long context (treat long context as a retrieval problem instead of raw attention).
- **Key insight:** naively extending context length degrades quality because position embeddings trained on shorter sequences don't generalize — the fix has to address the position-encoding scheme, not just increase a config value.

## LLMOps vs. Classical MLOps

- **Prompt versioning**: prompts are a first-class artifact that needs the same version control and rollback discipline as model weights — a prompt change can silently regress behavior as much as a model change.
- **Eval harnesses**: generative outputs need rubric-based or model-graded evaluation (LLM-as-judge, human eval) since there's no single scalar label like AUC.
- **Drift for generative systems**: monitoring shifts from feature/label drift to tracking output distribution changes, refusal rate changes, and user feedback signals (thumbs down rate, regeneration rate).
- **Cost as a first-class metric**: token cost per request is tracked and optimized alongside latency and quality — a lever classical ML serving didn't have.

## Mock Interview — LLM System Design (25 min)

**Prompt:** Design an AI coding assistant that reads a codebase and answers questions with citations, using tools (search, run tests) via MCP.

Cover, out loud, in order:
1. Clarify scope — repo size, languages, latency budget, whether it needs write access (run tests) or is read-only.
2. Architecture: retrieval over the codebase (embedding + ANN index vs. symbol-based search), agent loop with tool calls (search, read file, run tests) via MCP, context assembly, generation with citations to file:line.
3. Failure modes: stale index after a commit, agent looping on a failing test without making progress, hallucinated file paths — and mitigations for each.
4. Evaluation: how do you know an answer was "good"? (grounding check, does the cited code actually support the claim, does running tests reproduce the described behavior)
5. Cost/latency tradeoffs: when to skip the agent loop entirely (direct RAG for simple factual questions) vs. when multi-step tool use is justified.
