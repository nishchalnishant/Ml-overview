# Day 33-34: Alignment, Safety & Agentic Systems / RAG

## Day 33: Alignment & Safety

### RLHF vs. DPO

- **RLHF**: train a reward model on human preference comparisons (A preferred over B), then use PPO to fine-tune the LLM policy to maximize predicted reward, with a KL penalty against the SFT model to prevent drift too far from fluent language.
- **DPO**: reformulates the RLHF objective in closed form, directly optimizing the policy on preference pairs without a separate reward model or RL loop. Simpler and more stable to train.
- **Key tradeoff:** RLHF's separate reward model is reusable and inspectable but introduces **reward hacking** — the policy learns to exploit reward-model blind spots (e.g., verbosity, sycophancy) rather than genuinely improving. DPO avoids the extra model and instability of PPO but couples the reward signal directly into the policy, so you cannot adjust the reward function without retraining from the SFT checkpoint.

### Failure Modes

- **Reward hacking**: optimizing a proxy (reward model score) diverges from optimizing the true objective (human preference) — classic Goodhart's Law.
- **Sycophancy**: models trained on human preference data learn that agreeing with the user is often rewarded, independent of correctness.
- **Alignment tax**: safety/alignment tuning can measurably reduce capability on some benchmarks — a real tradeoff to name explicitly in interviews, not something to wave away.

### AI Safety & Responsible AI

- Red-teaming: adversarial prompting to surface harmful, biased, or jailbreak-prone outputs before deployment.
- Constitutional AI / RLAIF: using AI feedback (guided by a set of principles) instead of exclusively human labels to scale preference data generation.
- Production alignment failures are often subtler than headline jailbreaks — e.g., a model that's technically safe but confidently wrong, or one whose refusal behavior degrades task usefulness for legitimate requests.

## Day 34: Agentic Systems & RAG

### RAG Architecture

```
Query → (optional query rewriting) → Embedding
→ Vector DB ANN search (FAISS/Pinecone/pgvector)
→ top-k chunks → (optional cross-encoder reranker) → top-3
→ LLM generates answer grounded in retrieved context
→ citations extracted from source chunks
```

- **Failure modes by stage:** embedding (poor chunking, domain-mismatched embedding model), retrieval (low recall, no approximate index → latency), reranking (adds latency — skip for p50-sensitive paths, keep for p99 quality), generation (hallucination when context is insufficient or the model ignores it).
- **RAG vs. fine-tuning:** RAG when knowledge is dynamic and citations matter; fine-tuning when the task needs stable style/format adaptation or retrieval-hop latency is unacceptable.

### Hallucination Mitigation

- Grounding techniques: require citations tied to retrieved spans, and refuse or hedge when retrieval confidence is low rather than generating fluent-but-unsupported answers.
- Self-consistency / verification passes: sample multiple generations and check agreement, or run a separate verification prompt against retrieved evidence.

### Agentic Workflows

- An agent loop: LLM plans → selects a tool → executes → observes result → replans, iterating until task completion or a stop condition.
- Key design questions: how many tool-calling steps before forcing termination, how to handle tool errors gracefully, and how to prevent runaway loops or unsafe tool invocations.
- **MCP (Model Context Protocol)**: an emerging standard for exposing tools/data sources to LLM agents in a uniform way, replacing bespoke per-tool integration code — reduces the $N \times M$ integration problem (N models, M tools) to $N + M$.
